Weekly Webinars - Docker

What is Docker?

  • Docker is a software container platform
  • Fundamentally solves the “works on my machine” problem
  • Helps developers
    • Create and deploy apps fast
    • Define and package application needs
  • Uses the power of “containers”


What is a Container?

  • Standardized package including
  • Runs natively on the OS Kernel (without a full Operating System unlike Virtual Machines)
    • Docker is very lightweight in this regard
  • Flexible, Lightweight, Interchangeable, Portable, Scalable and Stackable



Containers vs. VMs



Installing and Running Docker

  • Install Docker Desktop from https://docs.docker.com/install/
    • Can be installed via Homebrew as well
  • Must have Docker Desktop installed on machine to run a Docker instance
  • Docker commands available from terminal / command line
    • Docker commands can be run from terminal with prefix ‘docker’
  • E.g. docker run hello-world
  • Creates local image repository for you to create and run images
  • Images can be downloaded and stored in an online repository like Docker Hub ( https://hub.docker.com/ ) 

docker run hello-world #runs a docker instance called hello-world

docker --help #gets descriptions of all items



Images, Containers, and Services

  • A docker image is an immutable snapshot of a live container
  • A container is an run-time instance of an image (either running or stopped)
  • Services run a single image, but codifies how the image runs including port numbers to use, number of container replicas to run, resource limits, etc.
  • Use a docker-compose.yml file to define, run, and scale services
  • Used in EBX env

Useful Docker Commands

docker pull <image:tag>
# Pulls a docker image from online repo to local repo

docker push <image:tag>
# Pushes modified image from local repo to online repo

docker build –t <tag> .
# Builds new docker image by running commands in ./Dockerfile

docker run –it <image>
# Run a docker image as a container interactively

docker images
# List images in local repo

$ docker ps
# List containers running locally



Continuous Integration Using Bitbucket Pipelines

  • Bitbucket Pipelines is built-in Continuous Integration/Continuous Development service
  • Provides automatic building, testing, and development of your code
  • Pipelines are run inside docker containers within the Bitbucket cloud infrastructure
  • Build configuration is defined in a file (bitbucket-pipelines.yml) that gets checked into the root of your repository


Advantages of Bitbucket Pipelines

  • Built-in to Bitbucket, so no separate build server required
  • Automatically scales on demand using Docker containers
  • Build configuration lives with code and is versioned with the code, making it easier to change how to build each project or branch (and pair with code)
  • Native support for Docker
  • Integration with AWS
  • Certain number of ‘build-minutes’ free with Bitbucket


Structure of a Pipeline File

  • pipelines: marks the beginning of all your pipeline definitions.
  • default: contains the steps that will run on every push.
  • step: each step starts a new Docker container that includes a clone of your repository, and then runs the contents of your script section inside it.
  • script: a list of commands that are executed in sequence.

Viewing Pipeline Builds in Bitbucket

There are several different ways to access the results of a repository’s pipeline builds from within Bitbucket:

  • From the “ Repositories ” page
  • From a repo’s “ Commits ” page
  • From a repo’s “ Branches ” page
  • From a repo’s “ Pipelines ” page



Build Artifacts and Downloads

Artifacts:

  • Defined in ‘artifacts’ section of pipeline
  • Mainly used to persist files between pipeline steps
  • Automatically deleted after 7 days
  • Before deletion, can be downloaded from the Artifacts section of the pipeline build

Downloads:

  • Persistent storage of build files
  • Downloadable from repo’s “Downloads” page
  • Copied to downloads section with curl command in pipeline
  • Bitbucket API can be used to access another repo’s downloads

Running Build Containers Manually

It is possible to run the docker container used in the pipeline outside of the pipeline. This can be used to:


  • test changes to the pipeline before committing 
docker run -it --volume=/Users/kendrick/w/ebx_docs:/ebx_docs --workdir="/ebx_docs" --memory=4g --memory-swap=4g --memory-swappiness=0 --entrypoint=/bin/bash guildone/sphinx:latest


  • spin up a pre-canned dev environment to build locally 
docker run -it --volume=/Users/kendrick/w/ebx_docs:/ebx_docs --workdir="/ebx_docs" --memory=4g --memory-swap=4g --memory-swappiness=0 guildone/sphinx:latest make –C docs clean html
